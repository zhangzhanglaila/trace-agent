# 04 · 开发路线图

原则：**分阶段、小步、可验证**。每完成一个子步骤更新本文件状态，并写入当日开发日志。
不一次性做完整个里程碑；每个子步骤应能独立跑通并验证。

状态图例：⬜ 未开始 · 🟡 进行中 · ✅ 完成

---

## 里程碑 M1 · 单次运行透明化（P0，当前重心）

目标：别人的 LangGraph Agent 跑一次，就能看清每一步并自动定位失败/卡点。
验收见 `01-产品需求.md` 的「P0 验收标准」。

| 步骤 | 范围 | 产出 | 状态 |
|------|------|------|:---:|
| M1.1 | 单次运行数据模型 | 定稿 `SingleRunReport` JSON 协议；从现有 TraceCapture 提取步骤时间线的构建器 | ✅ |
| M1.2 | 健康分析器 | `HealthAnalyzer`：定位失败步骤 / 卡点 / 慢步骤，产出 `health` 段 + 人话摘要 + 单测 | ✅ |
| M1.3 | LangChain 接入验证 | 用真实 LangGraph Agent 验证单次运行可被完整录制（不依赖第二次） | ✅ |
| M1.2.1 | 修 span 计时 | `end_span()` 重算 `latency_ms`，让慢步骤检测在真实 span 上生效（M1.3 发现的问题） | ✅ |
| M1.4 | 单次视图(自包含) | 零依赖 HTML 视图：状态徽章+诊断+时间线+高亮失败/卡点/慢+点击展开 | ✅ |
| M1.5 | 双通道启用 | `trace_run` 上下文 / `@observe` 装饰器一行出报告；导出 `step` 装饰器补精确步骤边界 | ✅ |
| M1.6 | 融入 Vue DevTools | 把单次视图做成 Vue DevTools 的一个标签页（需交互验证渲染） | ⬜ |

> 每步之间先验证再进入下一步。M1.1 的协议定稿是 M1.2/M1.4 的前置。

### 已知问题

- ✅ ~~span 每步耗时为 0~~ → M1.2.1 已修：`end_span()` 重算 latency_ms，真实 DeepSeek 运行
  下 LLM 步显示 861/820/694ms，回归测试全过。
- ⚠️ 待观察：根/父级 CHAIN 步骤包含子步骤耗时，理论上可能被慢步骤检测误判为离群。当前
  median×factor 阈值已能天然规避常见情况；若 M1.4 前端出现误标，再让 `HealthAnalyzer`
  排除容器步骤（需报告里带 parent 标记）。

## 里程碑 M2 · 实时监控运行中（P1）

依赖 M1 的步骤数据模型。目标：Agent 运行时实时推送步骤事件，前端像进度条一样展示。

| 步骤 | 范围 | 状态 |
|------|------|:---:|
| M2.1 | 步骤事件流（运行中逐步 emit） | TraceContext.on_step_end + trace_run.on_step 回调机制 + 5 项单测 | ✅ |
| M2.2 | 前端实时订阅与进度展示 | SSE 推送服务器 + 前端 EventSource 订阅 + 零依赖进度条演示 | ✅ |
| M2.3 | 卡住/超时实时告警 | `_TimeoutWatcher` 后台线程 + `push_alert_event` + 5 项单测 | ✅ |
