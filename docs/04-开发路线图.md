# 04 · 开发路线图

原则：**分阶段、小步、可验证**。每完成一个子步骤更新本文件状态，并写入当日开发日志。
不一次性做完整个里程碑；每个子步骤应能独立跑通并验证。

## 里程碑 M1 · 单次运行透明化（P0，当前重心）

目标：别人的 LangGraph Agent 跑一次，就能看清每一步并自动定位失败/卡点。
验收见 `01-产品需求.md` 的「P0 验收标准」。

| 步骤 | 范围 | 产出 | 状态 |
|------|------|------|:---:|
| M1.1 | 单次运行数据模型 | 定稿 `SingleRunReport` JSON 协议；从现有 TraceCapture 提取步骤时间线的构建器 | ✅ |
| M1.2 | 健康分析器 | `HealthAnalyzer`：定位失败步骤 / 卡点 / 慢步骤，产出 `health` 段 + 人话摘要 + 单测 | ✅ |
| M1.3 | LangChain 接入验证 | 用真实 LangGraph Agent 验证单次运行可被完整录制（不依赖第二次） | ✅ |
| M1.2.1 | 修 span 计时 | `end_span()` 重算 `latency_ms`，让慢步骤检测在真实 span 上生效（M1.3 发现的问题） | ⬜ |
| M1.4 | 前端单次视图 | UI 新增「单次运行」时间线视图，高亮失败/卡住步骤 | ⬜ |
| M1.5 | 双通道启用 | `enable()` 后单次运行自动出报告；`@trace` 装饰器补精确步骤边界 | ⬜ |

> 每步之间先验证再进入下一步。M1.1 的协议定稿是 M1.2/M1.4 的前置。

### 已知问题（M1.3 发现）

- **span 每步耗时为 0**：`agent_obs/trace_core.py` 的 `end_span()` 只回填输出/状态，未重算
  `latency_ms`，导致 LLM/工具等跨度步骤的单步耗时丢失，**慢步骤检测在真实运行上失效**。
  整段总耗时（首步 start→末步 end）不受影响。修复 = M1.2.1（小改，属录制内核，改后需跑
  `test_v2_compiler.py`、`test_explain_engine.py`）。
