# 03 · 设计规范

## 编码原则

1. **零侵入优先**：任何面向用户的接入，默认不要求改别人 Agent 代码。
2. **单次透明化不依赖 diff**：P0 能力独立于 A/B 对比，不引入两次运行的假设。
3. **协议先行**：跨前后端的数据结构先在 `02-技术架构.md` 定契约，再实现。
4. **小步提交**：一个 PR/一次提交只做一件可验证的事（对齐路线图的子步骤）。

## Python 规范

- 目标版本 Python 3.9+（见 `pyproject.toml`），不用 3.10+ 独有语法（如 `match`、`X | Y` 类型注解写在运行期）。
- 命名：模块/函数 `snake_case`，类 `PascalCase`，常量 `UPPER_SNAKE`。
- 新增公开 API 必须有 docstring，说明入参/返回/副作用。
- 分析器等纯逻辑模块**不得**直接依赖前端或 IO；保持可单测。

## 前端规范（agent-trace-ui）

- Vue 3 + `<script setup>` + TypeScript；状态走 Pinia（`traceStore.ts`）。
- 新增视图复用现有组件与协议类型（`src/types/trace.ts`），协议字段与后端 JSON 对齐。
- 「单次运行」视图是新增视图，不破坏现有 A/B 对比视图。

## 测试规范

- 每个新分析器（如 `HealthAnalyzer`）配单元测试，覆盖：正常、失败步骤、卡点、慢步骤四类。
- 测试文件放项目根或就近 `test_*.py`，沿用现有 pytest 约定。
- 提交前至少跑通相关测试；改动录制内核需跑 `test_v2_compiler.py`、`test_explain_engine.py`。

## Git 提交规范

- 提交信息格式：`<type>: <简述>`，type ∈ `feat/fix/docs/refactor/test/chore`。
- 不在未经用户要求时提交或推送。需要提交时先建分支（当前在 `main`）。
- 提交信息可用中文，保持与历史一致。

## 文档与日志规范

- 改动涉及需求/架构 → 同步更新 `docs/` 对应文件并在「变更记录」追加一行。
- 每个开发日在 `开发日志/` 记录完成事项与待办，见 `开发日志/README.md`。
