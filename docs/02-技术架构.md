# 02 · 技术架构

## 现有分层（已实现）

```
Agent Code
  → TracedAgent / adapters（录制入口）
  → TraceCapture（trace_core.py：录制每步细节）
  → ExecutionGraph（execution_graph.py：SSA + CFG + φ-节点 中间表示）
  → TraceDiffer（trace_diff.py：两图对齐 diff，A/B 根因）
  → frontend_adapter.py（Trace → 统一 JSON 协议）
  → agent-trace-ui（Vue 3 前端）
```

关键文件（改动前必读）：

| 文件 | 职责 |
|------|------|
| `agent_obs/trace_core.py` | `TracedAgent` / `TraceContext` / 语义类型 / 录制内核 |
| `agent_obs/execution_graph.py` | 执行图 IR（SSA/CFG/φ），到达定义分析 |
| `agent_obs/trace_diff.py` | A/B 对比与因果判定 |
| `agent_obs/frontend_adapter.py` | Trace → 前端 JSON 协议 |
| `agent_obs/enable.py` | `enable()` / `dev()` 入口 |
| `agent_obs/adapters/langchain.py` | LangChain 自动 patch 适配器 |
| `agent_obs/cli_main.py` | CLI：run / diff / debug |
| `agent-trace-ui/` | Vue 3 前端 + `server.py` API |

## P0 新增：单次运行透明化架构

复用现有录制内核（TraceCapture 已经在录单次运行的数据），**新增分析与呈现层**，不改录制协议。

```
（已有）TraceCapture 单次录制
  → [新] SingleRunReport 构建器：从 trace 提取步骤时间线
  → [新] HealthAnalyzer：定位 失败步骤 / 卡点 / 慢步骤
  → SingleRunReport JSON（新协议，见下）
  → frontend_adapter：输出单次视图数据
  → [新] 前端「单次运行」时间线视图
```

设计原则：**单次透明化不依赖第二次运行**，也不依赖 diff 引擎。A/B 对比仍走原有 TraceDiffer 路径。

### 单次运行报告协议（M1.1 定稿 v1）

由 `agent_obs/single_run.py` 的 `build_single_run_report()` 产出。字段映射自现有
`TraceCapture.steps` 的真实字段，构建器为**纯逻辑、无 IO**。

```jsonc
{
  "run_id": "string",
  "run_name": "agent_run",
  "status": "success | failed | unknown",   // M1.1：有 error 即 failed。stuck 由 M1.2 补
  "started_at": 0.0,                          // 首步 start_time（epoch 秒），无则 null
  "duration_ms": 0.0,                         // 末步 end - 首步 start；退化为各步 latency 之和
  "step_count": 0,
  "steps": [
    {
      "id": "tool_2",
      "name": "weather_api",                  // semantic_name > name > var > type
      "kind": "llm | tool | branch | function | output | merge",  // 映射自 step.type
      "status": "ok | error | running",       // error(有 error) / running(起了未结束) / ok
      "input": {},                            // prompt / args / inputs / condition
      "output": {},                           // output / result / outputs / value
      "duration_ms": 0.0,                     // 映射自 latency_ms
      "error": null                           // 有异常时填错误信息
    }
  ],
  "health": {                                 // M1.1 占位，M1.2 由 HealthAnalyzer 填
    "failed_step_id": null,                   // 抛异常的步骤
    "stuck_step_id": null,                    // 最后执行到、未完成的步骤（卡点）
    "slow_step_ids": [],                      // 耗时异常步骤
    "summary": ""                             // 人话一句：卡在了哪、为什么
  }
}
```

**字段映射规则**（`step.type` → `kind`）：`llm→llm`、`tool→tool`、`branch→branch`、
`chain→function`、`output→output`、`merge→merge`。
`status`：优先看 `error` 字段 → `error`；llm/tool 起了但 `output/result` 仍为 None 且无 error → `running`（疑似卡住，M1.2 判定）；否则 `ok`。

> 协议为 v1；M1.2 会填充 `health` 并可能把顶层 `status` 扩展出 `stuck`。前后端据此对齐。
